# ADR-0075: GLM-5.2 price drop — consolidate GLM-5/GLM-5.1 catalog and branded aliases onto GLM-5.2

**Status:** Accepted
**Date:** 2026-07-02
**Deciders:** @stephane-segning
**Relates to:** amends the GLM-5.2 non-adoption call in [ADR-0050](0050-kimi-k2.7-code-supersedes-k2.6-as-frontend-pro-backing.md) (the Kimi K2.7 `adorsys-frontend-pro` backing decision in that ADR is unaffected); refines the branded-alias backings from [ADR-0044](0044-opencode-role-subagents-and-permission-scoped-tools.md)

## Context

ADR-0050 (2026-06-18) evaluated GLM-5.2 on DeepInfra at $1.40 / $0.25 / $4.40
per 1M (input/cached/output) and declined to adopt it: at that price its
intelligence-per-dollar was only equal to GLM-5.1, making it a premium tier
rather than a better-value replacement for the existing GLM-5/GLM-5.1
backings.

Re-verified live against DeepInfra on 2026-07-02, GLM-5.2's price has dropped
to **$0.93 / $0.18 / $3.00** per 1M — a 33–40% cut across all three axes. This
now **strictly undercuts GLM-5.1** ($1.05 / $0.205 / $3.50) on every axis
while offering a 1,048,576-token context (vs GLM-5.1's 202,752) and a
materially higher AA Intelligence Index (51 vs GLM-5.1's 40, per ADR-0050's
own citation). It prices above plain GLM-5 ($0.60 / $0.12 / $2.08) on raw
per-token cost, but the ~5x larger context and reasoning uplift make it the
clear pick over both older tiers.

## Decision

1. **Repoint every catalog entry currently backed by GLM-5 or GLM-5.1 to
   GLM-5.2** (`zai-org/GLM-5.2` on DeepInfra), at $0.93 / $0.18 / $3.00 per
   1M, 1,048,576-token context:
   - `reviewer` (was GLM-5)
   - `adorsys-planner` (was GLM-5)
   - `adorsys-planner-pro` (was GLM-5.1)
   - `adorsys-coder-pro` (was GLM-5.1)
   - `adorsys-reviewer-pro` was already GLM-5.2 — only its `inputPer1M` is
     corrected from $0.95 to the confirmed $0.93 (cached/output were already
     accurate).
2. **Consolidate the two standalone raw catalog ids `glm-5` and `glm-5p1`
   into a single `glm-5p2` id**, same GLM-5.2 backing/pricing. The LibreChat
   default `modelSpecs` catalog list is updated accordingly.
3. **Do not retune the `adorsys-*` per-model budget caps** as part of this
   change — a separate, deferred follow-up (tracked informally; a
   1M-context model under the existing budget deserves its own look once
   real usage exists, not a pre-emptive change).

## Consequences

**Positive**
- Every catalog id that used to route to GLM-5/GLM-5.1 gets a strictly
  larger context ceiling and (per ADR-0050's own AA Intelligence Index
  citation) a materially stronger model, at a lower or comparable price.
- Two redundant raw catalog ids (`glm-5`, `glm-5p1`) collapse into one
  (`glm-5p2`), shrinking the model list clients see.

**Negative**
- `adorsys-planner` and `adorsys-planner-pro` now share an identical
  backing model — the value/pro split for the planner brand becomes a
  no-op until a future premium tier is introduced. Flagged, not blocking:
  the two ids stay because they're independent budget buckets
  (ADR-0044) and agent-pinned in different places.
- The `glm-5` / `glm-5p1` ids disappear from the catalog. A user who had
  explicitly selected either (not a brand) must reselect `glm-5p2` — same
  acceptable-migration precedent as ADR-0050's `kimi-k2.6` retirement.

**Neutral / follow-ups**
- Revisit `adorsys-*` budget caps once GLM-5.2's 1M context sees real usage
  (per the deferred note above).
- Docs updated: `docs/architecture/09-model-serving.md` (branded-alias
  backing example), `docs/per-user-observability.md` and
  `docs/opencode-well-known.md` (runnable examples using the retired raw
  ids). `docs/2026-06-08-gpu-platform-procurement-comparison.md` and
  `docs/2026-hetzner-cutover.md` are point-in-time snapshots and are left
  as historical record.

## Alternatives considered

- **Keep GLM-5/GLM-5.1 alongside GLM-5.2** — rejected: GLM-5.2 now
  dominates GLM-5.1 on every price axis with a much larger context, so
  keeping GLM-5.1 in the catalog would only be dead weight. Plain GLM-5
  is genuinely cheaper per-token, but the maintainer's call is to
  standardize the catalog on the stronger model rather than keep a
  budget/quality split at this price gap.
- **Add `glm-5p2` as a new id and leave `glm-5`/`glm-5p1` in place** —
  rejected: keeps two functionally-superseded, near-duplicate raw ids
  in the user-facing catalog for no benefit.
