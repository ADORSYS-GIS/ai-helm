# ADR-0017: Workloads target the home-remote cluster, never in-cluster (guarded)

**Status:** Accepted
**Date:** 2026-05-31
**Deciders:** @stephane-segning

## Context

Three templates in this repo emit ArgoCD `Application` /
`ApplicationSet` destinations:

| Template | Pre-ADR destination |
|---|---|
| `charts/apps/templates/applications.yaml` | **hardcoded** `name: lke560142-ctx` |
| `charts/ai-models/templates/applicationset.yaml` | `.Values.argocd.destination.name`, defaulting to **`in-cluster`** |
| `charts/librechart/templates/applicationset.yaml` | `.Values.argocd.destination.name`, defaulting to **`in-cluster`** |

The target cluster is registered in ArgoCD under the explicit **name
`home-remote`**. The GitOps entrypoint (the root Application in
`ai-gitops`) already uses it:

```yaml
# root Application — lives in ai-gitops, points at charts/apps here
- name: ai-apps-v2
  project: ai
  source:
    repoURL: https://github.com/ADORSYS-GIS/ai-helm
    targetRevision: <branch>
    path: charts/apps
  destination:
    name: home-remote        # ← the registered cluster name
    namespace: argocd
  syncPolicy:
    automated: { prune: true, selfHeal: true }
```

So `charts/apps` itself is reconciled onto the `home-remote` cluster, the
~37 Application CRDs it emits land in that cluster's `argocd` namespace,
and ArgoCD there reconciles each of them. Every one of those generated
Applications must reference the **same** cluster the same way — by the
registered name `home-remote`.

Two problems with the pre-ADR state:

1. **Inconsistency.** `apps` hardcoded `lke560142-ctx`; the two
   orchestrators defaulted to a *different* handle, `in-cluster`. So a
   child Application generated by `ai-models` or `librechart` would
   reference the cluster by ArgoCD's built-in in-cluster handle instead
   of the registered `home-remote` name — a different (and here,
   unregistered/likely-disabled) destination than the entrypoint and the
   rest of the fleet use.
2. **No safety net.** Nothing stopped a destination from resolving to
   ArgoCD's built-in in-cluster reference (`name: in-cluster` or
   `server: https://kubernetes.default.svc`). Whether or not that handle
   happens to resolve to the same physical API server as `home-remote`,
   it is **not** how this repo addresses the cluster — the named
   registration (`home-remote`) is, and mixing the two splits one
   cluster into two ArgoCD destinations with different credentials/config.

> **Note on topology.** This ADR does *not* assume `home-remote` is a
> physically separate cluster from where ArgoCD runs — it may well be the
> same cluster, registered under a friendly name rather than addressed via
> the `in-cluster` default. The invariant is about **how** the cluster is
> referenced (always the registered name, never the built-in in-cluster
> handle), not about machine topology.

ADR-0006 already flagged the hardcoded `lke560142-ctx` and listed
"parameterize `destination.name`" as a migration item. This ADR
addresses the destination-control slice directly (independent of the
larger multi-source AppSet migration ADR-0006 still proposes).

## Decision

> **Correction (2026-05-31, same PR):** the original single-tier framing
> below — "every Application targets `home-remote`" — was wrong for
> ArgoCD **control objects**. An `Application` / `ApplicationSet` CR must
> live on the cluster ArgoCD runs in, in the `argocd` namespace, or the
> controllers never see it. So the model is **two-tier** (see "Control
> objects vs workloads" below). The home-remote invariant + the guard
> still apply to **workloads**.

**Invariant (workloads):** every Application that deploys **workloads**
references the target cluster by its registered name **`home-remote`**.
Using ArgoCD's built-in in-cluster handle (`name: in-cluster` /
`server: https://kubernetes.default.svc`) for a workload is forbidden
unless explicitly opted in (`allowInCluster: true`).

### Control objects vs workloads (two-tier)

ArgoCD has two distinct "where" questions:

1. **Where the control object lives** — `Application` / `ApplicationSet`
   CRs are watched by the ArgoCD controllers in the `argocd` namespace on
   the cluster ArgoCD runs in. They must be created **in-cluster**.
2. **Where the workload deploys** (`spec.destination`) — the target for
   the actual Deployments/Services/etc. → **`home-remote`**.

Mapping that onto this repo's charts:

| Chart emits… | Its Application's `spec.destination` |
|---|---|
| Workloads (mimir, grafana, core-gateway, leaf charts, …) | `home-remote` |
| A control object — an **ApplicationSet** (orchestrators `ai-models`, `librechart`) | **`server: https://kubernetes.default.svc`** / `argocd` |

So in `charts/apps`, an app whose deployed content is itself a control
object sets **`controlPlane: true`** on its entry; the template then
targets `inClusterServer` (**`server: https://kubernetes.default.svc`**) /
`controlPlaneNamespace` (`argocd`) and bypasses the workload guard. We use
the in-cluster **server URL** rather than the `name: in-cluster` handle:
the URL is always recognised as the local cluster, whereas the name
depends on that cluster being registered under it. The orchestrators'
ApplicationSet `template.spec.destination` (the **child** Applications)
stays `home-remote` — children deploy workloads. And `charts/apps` itself
is deployed by the root `ai-apps-v2` Application, whose destination must
also be `server: https://kubernetes.default.svc` so the generated
Application CRs land where the local controller watches them (that lives
in `ai-gitops`).

### One controllable knob per emitting chart

Each of `apps`, `ai-models`, `librechart` exposes the same
`argocd.destination` block (this is the **workload** destination — for the
orchestrators it's the destination of the child Applications):

```yaml
argocd:
  destination:
    name: home-remote        # cluster context registered in ArgoCD (verbatim → spec.destination.name)
    server: ""               # alternative to name → spec.destination.server (a cluster API URL)
    allowInCluster: false    # escape hatch
    namespace: <ns>          # (ai-models / librechart) or per-app (apps)
```

`charts/apps` additionally exposes `argocd.inClusterServer` (default
`https://kubernetes.default.svc`) + `argocd.controlPlaneNamespace`
(default `argocd`) for the `controlPlane: true` apps.

The canonical home-remote cluster identity is the literal name
**`home-remote`** — a cluster registered under that name in ArgoCD, used
verbatim as `spec.destination.name`. This replaces the old
`lke560142-ctx` magic string.

### One controllable knob per emitting chart

Each of `apps`, `ai-models`, `librechart` exposes the same
`argocd.destination` block:

```yaml
argocd:
  destination:
    name: home-remote        # cluster context registered in ArgoCD (verbatim → spec.destination.name)
    server: ""               # alternative to name → spec.destination.server (a cluster API URL)
    allowInCluster: false    # escape hatch
    namespace: <ns>          # (ai-models / librechart) or per-app (apps)
```

The canonical home-remote cluster identity is the literal name
**`home-remote`** — a cluster registered under that name in ArgoCD, used
verbatim as `spec.destination.name`. This replaces the old

### Hard-fail guard

A shared helper —
`<chart>.argocd.destinationClusterRef` in each chart's
`templates/_helpers.tpl` — emits the cluster identity line (`name:` or
`server:`) and **fails the Helm render** with `fail` if the destination
resolves to the in-cluster API:

- `name == "in-cluster"`, or
- `server ∈ { https://kubernetes.default.svc, https://kubernetes.default.svc:443 }`

…unless `allowInCluster: true` is set. The failure message names the
offending values and points here.

Because the guard runs at template time, the failure surfaces in
`helm template`, in CI's chart-render check, AND when ArgoCD itself
renders the chart — three independent gates, all before anything syncs.

### Why `fail` (not silent normalization)

A render failure is loud and actionable. Silently rewriting an
in-cluster destination to `home-remote` would hide a real
misconfiguration (someone *thought* the built-in in-cluster handle would
be used) and remove the operator's ability to ever legitimately use it.
The `allowInCluster` flag is the deliberate, greppable opt-in for the
rare case where a component must be addressed via the in-cluster handle
rather than the registered `home-remote` name (e.g. a bootstrap
component applied before the named cluster registration exists).

## Consequences

**Positive**
- Single source of truth for the destination cluster per chart; no more
  hardcoded `lke560142-ctx`.
- An accidental in-cluster destination can't reach a cluster — it breaks
  the render first.
- The control surface is uniform across all three emitters: same value
  names, same helper semantics, same escape hatch.
- `server`-based destinations (URL) are supported for the
  multi-cluster-by-URL case without changing the contract.

**Negative**
- The guarantee is **render-time**, not admission-time. A destination
  written directly into a hand-authored Application in `ai-gitops`
  (bypassing these charts) is not covered — those live in a different
  repo. Mitigation: this repo is the chart source; `ai-gitops` overrides
  values, and values flow through the guard. A truly belt-and-suspenders
  control would be an ArgoCD `AppProject` `destinations:` allowlist
  restricting the project to the home-remote cluster (see follow-ups).
- Three near-identical helper copies (one per chart) — Helm has no clean
  cross-chart helper sharing without a library dependency, and these
  three charts don't otherwise share one. Drift risk is low (the helper
  is ~15 lines and stable); the copies are documented as siblings.

**Neutral / follow-ups**
- **AppProject allowlist.** The `ai` `AppProject` (in `ai-gitops`) should
  also pin `spec.destinations[].server`/`name` to the home-remote
  cluster. That's the admission-time complement to this render-time
  guard and belongs in the deployment-state repo. Tracked for a
  follow-up there.
- When a second workload cluster appears, `server`/`name` becomes a
  per-element value (Matrix generator) — the helper already accepts
  either form, so only the values shape changes. ADR-0006's "successor
  ADR when a second cluster lands" still applies.

## Alternatives considered

- **Silent normalize in-cluster → home-remote.** Hides
  misconfiguration; no opt-in path. Rejected (see "Why fail").
- **Warn-only (Helm NOTES).** Doesn't actually prevent in-cluster;
  NOTES are easy to miss and don't show in ArgoCD's render. Rejected.
- **AppProject destination allowlist only (no render guard).** The
  correct admission-time control, but it lives in `ai-gitops` and
  doesn't catch the mistake at author time / in this repo's CI. Best as
  a *complement*, not a replacement. Adopted as a follow-up, not instead.
- **A shared library chart for the helper.** DRY, but adds a chart
  dependency to all three emitters for ~15 lines. Rejected as
  over-engineering for the current count; revisit if a fourth emitter
  appears.
