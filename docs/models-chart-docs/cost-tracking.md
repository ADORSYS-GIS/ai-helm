# Cost Tracking And Budgeting

This file is the plain-English guide to how rate limiting works in this chart.

If you only remember four things, remember these:

1. The monthly limit is based on estimated model cost, not raw request count.
2. The estimate comes from the model prices in `charts/ai-models/values.yaml`.
3. Input tokens, cached input tokens, and output tokens can cost different amounts.
4. The request that crosses the monthly budget can still succeed, because the budget is reduced after the response comes back.

## The Two Things You Configure

Rate limiting in this chart is controlled by two groups of values:

| Where | What it controls | Plain-English meaning |
| --- | --- | --- |
| `rateLimitBudgeting.plans` | Monthly budget per billing plan | "How much estimated spend can this plan use per month?" |
| `models.<name>.pricing` | How request cost is calculated for one model | "How expensive are this model's input, cached input, and output tokens?" |

## Simple Glossary

| Term | Meaning |
| --- | --- |
| Input tokens | Tokens sent in the prompt. |
| Cached input tokens | Prompt tokens the provider says were served from cache. These are usually cheaper. |
| Output tokens | Tokens generated by the model. These are usually more expensive. |
| Standard pricing | The normal price tier for a model. |
| Long-context pricing | A higher price tier used by some models when the input prompt is very large. |
| Micro-USD | One millionth of a US dollar. Envoy uses integers, so the chart stores request cost in this unit. |

## Why We Do Not Treat All Tokens The Same

Not all tokens cost the same to the provider:

1. Fresh prompt tokens are usually cheaper than generated output tokens.
2. Cached prompt tokens are often discounted even more.
3. Some models charge more when the prompt crosses a large context threshold.

That is why the chart does not just say "1 token = 1 unit". It turns token usage into an estimated cost that better matches provider pricing.

## How To Read `pricing` In `values.yaml`

Each model has a `pricing` block under `models.<name>.pricing`.

There are three supported strategies.

### `weighted`

Use `weighted` when the provider publishes separate prices for:

1. uncached input tokens
2. cached input tokens
3. output tokens

Example:

```yaml
pricing:
  strategy: weighted
  standard:
    inputPer1M: 0.75
    cachedInputPer1M: 0.075
    outputPer1M: 4.50
```

This means:

1. 1,000,000 fresh input tokens cost `$0.75`
2. 1,000,000 cached input tokens cost `$0.075`
3. 1,000,000 output tokens cost `$4.50`

### `flat`

Use `flat` when the provider publishes one blended price for total tokens and does not split input and output prices in a useful way.

Example:

```yaml
pricing:
  strategy: flat
  standard:
    effectivePer1M: 0.20
```

This means all tokens are treated the same for budgeting purposes.

### `tieredWeighted`

Use `tieredWeighted` when the provider has both:

1. separate input, cached input, and output prices
2. a more expensive long-context tier after a token threshold

Example:

```yaml
pricing:
  strategy: tieredWeighted
  thresholdTokens: 200000
  standard:
    inputPer1M: 1.25
    cachedInputPer1M: 0.125
    outputPer1M: 10.00
  longContext:
    inputPer1M: 2.50
    cachedInputPer1M: 0.25
    outputPer1M: 15.00
```

This means:

1. if `input_tokens <= 200000`, use `standard`
2. if `input_tokens > 200000`, use `longContext`

## The Math In Plain English

### For `weighted`

The chart first calculates fresh input tokens:

```text
uncached_input_tokens = max(input_tokens - cached_input_tokens, 0)
```

Then it calculates estimated request cost:

```text
estimated_cost_micro_usd =
  (uncached_input_tokens * inputPer1M) +
  (cached_input_tokens * cachedInputPer1M) +
  (output_tokens * outputPer1M)
```

Why does this produce `micro_usd`?

Because the prices are stored as "USD per 1,000,000 tokens". Multiplying the token count by that number gives the same numeric result as micro-USD.

Example:

1. Pricing: `input=0.75`, `cached=0.075`, `output=4.50`
2. Usage: `input_tokens=10000`, `cached_input_tokens=9000`, `output_tokens=100`
3. Fresh input tokens: `10000 - 9000 = 1000`
4. Cost:
   `1000 * 0.75 + 9000 * 0.075 + 100 * 4.50 = 1875`
5. Final value stored in metadata: `1875` micro-USD
6. In normal dollars: `$0.001875`

### For `flat`

The chart ignores token direction and uses one all-in rate:

```text
estimated_cost_micro_usd = total_tokens * effectivePer1M
```

Example:

1. Pricing: `effectivePer1M=0.20`
2. Usage: `total_tokens=20000`
3. Cost: `20000 * 0.20 = 4000` micro-USD
4. In normal dollars: `$0.004`

### For `tieredWeighted`

The chart first decides which pricing tier to use:

```text
if input_tokens > thresholdTokens:
  use longContext
else:
  use standard
```

Then it applies the same `weighted` formula with the chosen tier.

Example:

1. Threshold: `200000`
2. Request usage: `input_tokens=250000`, `cached_input_tokens=0`, `output_tokens=1000`
3. Since `250000 > 200000`, the chart uses `longContext`
4. If `longContext` prices are `2.50 / 0.25 / 15.00`, the cost is:
   `250000 * 2.50 + 0 * 0.25 + 1000 * 15.00 = 640000`
5. Final value stored in metadata: `640000` micro-USD
6. In normal dollars: `$0.64`

## Why The Rendered CEL Looks More Complicated

The simple formulas above are the right way to think about the pricing model.

The CEL rendered into `AIGatewayRoute` looks more complex for two practical reasons:

1. Envoy CEL needs integer-safe math for this use case.
2. The token counters are emitted as unsigned integers, so the chart converts them before subtracting cached tokens.

So the chart does this internally:

1. multiply each `*Per1M` price by `1000`
2. round it to an integer in Helm
3. do the token math in CEL with integers
4. divide by `1000` at the end

For example, this human-friendly pricing:

```yaml
pricing:
  strategy: weighted
  standard:
    inputPer1M: 0.75
    cachedInputPer1M: 0.075
    outputPer1M: 4.50
```

is rendered roughly as:

```cel
((max((int(input_tokens) - int(cached_input_tokens)), 0) * 750) +
 (int(cached_input_tokens) * 75) +
 (int(output_tokens) * 4500)) / 1000
```

That looks more technical, but it is still the same pricing rule:

1. fresh input tokens use the input rate
2. cached input tokens use the cached rate
3. output tokens use the output rate

## What Envoy Does With That Number

The flow is:

1. `AIGatewayRoute.spec.llmRequestCosts` asks Envoy AI Gateway to emit token counters and `llm_custom_total_cost`.
2. Envoy AI Gateway stores them as dynamic metadata under `io.envoy.ai_gateway`.
3. `BackendTrafficPolicy` reads `io.envoy.ai_gateway/llm_custom_total_cost`.
4. Envoy subtracts that response cost from the monthly budget bucket for the matching account, plan, and model.

The important detail is step 4 happens on the response path.

That means:

1. the request that crosses the monthly budget can still succeed
2. the next matching request is the one that gets blocked once Redis already contains the exhausted bucket

This delayed enforcement is why the chart also supports a simple requests-per-minute fallback.

## Who Gets Limited

There are two different rules in `BackendTrafficPolicy`.

### Monthly budget rule

This rule matches on:

1. `x-account-id`
2. `x-billing-plan`
3. `x-ai-eg-model`

Meaning:

1. each account gets its own budget bucket
2. the bucket depends on the billing plan
3. the bucket is separate for each model

### Fallback burst rule

This rule matches on:

1. `x-api-key-id`
2. `x-ai-eg-model`

Meaning:

1. each API key gets its own burst limit
2. the burst limit is separate for each model
3. this rule protects against spikes, not monthly spend fairness

## The Main Rate-Limit Settings In `values.yaml`

| Setting | What it means | Example |
| --- | --- | --- |
| `rateLimitBudgeting.plans.<plan>.monthlyBudgetUsd` | Default budget for all models | `30` |
| `rateLimitBudgeting.plans.<plan>.modelBudgets.overrides.<model>` | Per-model budget override | `gpt-5-mini: 10` |
| `models.<name>.pricing.strategy` | Which pricing formula to use | `weighted`, `flat`, `tieredWeighted` |
| `models.<name>.pricing.standard.inputPer1M` | Fresh input token price | `0.75` |
| `models.<name>.pricing.standard.cachedInputPer1M` | Cached input token price | `0.075` |
| `models.<name>.pricing.standard.outputPer1M` | Output token price | `4.50` |
| `models.<name>.pricing.standard.effectivePer1M` | Flat total-token price | `0.20` |
| `models.<name>.pricing.thresholdTokens` | Threshold that activates `longContext` pricing | `200000` |
| `models.<name>.pricing.longContext.*` | Prices used after the threshold is crossed | see Gemini profiles |

### Per-Model Budget Configuration

Each billing plan supports per-model budget overrides. The resolution order is:

1. Check `rateLimitBudgeting.plans.<plan>.modelBudgets.overrides.<model>`
2. Fall back to `rateLimitBudgeting.plans.<plan>.monthlyBudgetUsd`

Example configuration:

```yaml
rateLimitBudgeting:
  plans:
    free:
      monthlyBudgetUsd: 30            # Default: $30 per month for all models
      modelBudgets:
        overrides:
          gpt-5-mini: 10             # Expensive model: $10
          gemini-2.5-pro: 50         # Popular model: $50
```

This allows fine-grained control over spend allocation across different models while maintaining a sensible default.

## Where The Math Lives In The Chart

If you need to trace the implementation:

1. `charts/ai-models/values.yaml` holds the pricing data.
2. `charts/ai-models/templates/_helpers.tpl` builds the CEL expressions.
3. `charts/ai-models/templates/aigatewayroute.yaml` emits `llm_custom_total_cost`.
4. `charts/ai-models/templates/backendtrafficpolicy.yaml` uses that metadata as response cost.

## Practical Rules For Updating A Model

Use these rules when adding or changing a model:

1. Choose `weighted` if the provider gives different prices for fresh input, cached input, and output.
2. Choose `flat` if the provider only gives one practical token price.
3. Choose `tieredWeighted` if the provider also has a long-context tier.
4. Update the model's `pricing` block in `values.yaml`.
5. Do not change `BackendTrafficPolicy` just to refresh prices. That template already reads `llm_custom_total_cost`.

## Important Limitations

These limits should be understood by anyone operating the chart:

1. This is an estimated spend guardrail, not a billing-grade invoice engine.
2. Cached-input discounting only works when the provider reports cached token usage.
3. Image models can still be approximate because providers may publish separate text-input and image-input prices while the current Envoy metadata only gives aggregate `input_tokens` and `output_tokens`.
4. Very small requests can round down to `0` micro-USD because Envoy CEL uses integer math.
5. The metadata key must stay exactly `io.envoy.ai_gateway/llm_custom_total_cost`, or the monthly budget rule will stop working.
